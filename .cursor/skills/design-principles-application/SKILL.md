# Design Principles Application

## いつ使うか
- 設計判断の根拠を明確にしたい時
- リファクタリング方針を決める時
- パターン導入の是非を評価する時

## 参照ルール（最優先）
- `.cursor/rules/00-mindset.mdc`
- `.cursor/rules/40-architecture-and-layering.mdc`

## 要点（統合）
- SOLIDを「境界と責務」の判断基準にする
- デザインパターンは複雑性を下げる時のみ使う
- 開発法則はトレードオフの説明に使う

## 手順
1. 目的と制約を明文化する
2. SOLIDのどれが効くかを整理する
3. パターン適用で複雑性が下がるか評価する
4. 法則（Leaky Abstractions等）でリスクを説明する

## チェックリスト
- 単一責任が崩れていない
- 変更点が局所化されている
- 抽象が漏れている箇所を把握している
- パターン導入が目的化していない

## 出力テンプレ
1. 判断対象
2. 適用する原則/法則
3. パターン導入の是非
4. リスクとトレードオフ
