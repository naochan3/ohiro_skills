# System Design & ADR

## いつ使うか
- 事前設計やスケーラビリティ検討が必要な時
- アーキテクチャ判断を記録したい時
- クラウドネイティブ設計に合わせたい時

## 参照ルール（最優先）
- `.cursor/rules/30-non-functional-requirements.mdc`
- `.cursor/rules/28-observability-and-logging.mdc`

## 要点（統合）
- 要件はSLO/負荷/データ量の想定から始める
- ボトルネックは計算と通信の両面で見積もる
- 重要な設計判断はADRとして記録する
- 12-Factorは運用しやすさの基準になる

## 手順
1. 想定トラフィックとデータ量を概算
2. ボトルネックと拡張ポイントを特定
3. 主要な判断をADRで記録
4. 12-Factor準拠チェックを行う

## チェックリスト
- 目標SLOが明記されている
- ストレージ/キャッシュ/キューの選定理由がある
- ADRが残されている
- 設定/ログ/依存の分離ができている

## 出力テンプレ
1. 前提（負荷/データ/成長見込み）
2. 主要アーキテクチャ判断
3. ADR一覧
4. 12-Factor準拠チェック結果
